Deploy OpenClaw on Railway with a one-click template and access it through the web Control UI.
This is the easiest "no terminal on the server" path: Railway runs the Gateway for you.

## Quick checklist (new users)

1. Click **Deploy on Railway** (below).
2. Add a **Volume** mounted at `/data`.
3. Set the required **Variables** (at least `OPENCLAW_GATEWAY_PORT` and `OPENCLAW_GATEWAY_TOKEN`).
4. Enable **HTTP Proxy** on port `8080`.
5. Open `https://<your-railway-domain>/openclaw` and connect using your `OPENCLAW_GATEWAY_TOKEN`.

## One-click deploy

<a href="https://railway.com/deploy/clawdbot-railway-template" target="_blank" rel="noreferrer">
  Deploy on Railway
</a>

After deploy, find your public URL in **Railway → your service → Settings → Domains**.

Railway will either:

- give you a generated domain (often `https://<something>.up.railway.app`), or
- use your custom domain if you attached one.

Then open:

- `https://<your-railway-domain>/openclaw` — Control UI

## What you get

- Hosted OpenClaw Gateway + Control UI
- Persistent storage via Railway Volume (`/data`) so config/credentials/workspace survive redeploys

## Required Railway settings

### Public Networking

Enable **HTTP Proxy** for the service.

- Port: `8080`

### Volume (required)

Attach a volume mounted at:

- `/data`

The container entrypoint adjusts ownership of `/data` so the `node` user can write
`OPENCLAW_STATE_DIR` (for example `/data/.openclaw`). Without that step, a root-owned
volume causes `EACCES` when saving config.

### Variables

Set these variables on the service:

- `OPENCLAW_GATEWAY_PORT=8080` (required — must match the port in Public Networking)
- `OPENCLAW_GATEWAY_TOKEN` (required; treat as an admin secret)
- `OPENCLAW_STATE_DIR=/data/.openclaw` (recommended)
- `OPENCLAW_WORKSPACE_DIR=/data/workspace` (recommended)

### Health check

Railway probes your public **HTTP** port. The gateway must listen on all interfaces on that same port, and the check path should be **`/healthz`** (or `/health`).

If deploy succeeds but **Network → Healthcheck** fails, verify:

- `OPENCLAW_GATEWAY_PORT` matches the port Railway is routing to (often `8080`, or the same value as Railway `PORT` when you wire them together).
- The container start command includes **`--bind lan`** (the default `Dockerfile` `CMD` includes this) so the process listens on `0.0.0.0`, not only loopback.
- `OPENCLAW_GATEWAY_TOKEN` is set so the gateway can safely bind beyond loopback.

### Memory (1 GiB and other small plans)

The gateway loads a large TypeScript bundle and optional services; **about 2 GiB RAM** is comfortable for production. On **1 GiB** (or smaller) Railway plans you may see **out of memory** crashes.

Do this first:

- The published image sets **`NODE_OPTIONS=--max-old-space-size=512`** so the JS heap does not grow until Linux kills the container. If you override **`NODE_OPTIONS`** in Railway, include the same flag (merge your extra flags into one value).
- Optionally set **`OPENCLAW_SKIP_CANVAS_HOST=1`** to turn off the separate Canvas file server and save some RAM if you do not need it.

If it still OOMs, increase the service **memory limit** in Railway (Settings → Resources) or reduce load (fewer channels / lighter usage).

## Backups & migration

Export your configuration and workspace:

```bash
openclaw backup create
```

This creates a portable backup archive you can restore on any OpenClaw host. See [Backup](/cli/backup) for details.
